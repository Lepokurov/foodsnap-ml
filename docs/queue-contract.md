# Queue Contract

This document describes the RabbitMQ contract between the FoodSnap ML API service and external worker microservices.

## Decision

The API service is a producer only.

It does not consume meal-analysis jobs and does not run an embedded worker inside the FastAPI lifespan.

External worker microservices are responsible for:
- consuming RabbitMQ messages
- loading required state from `PostgreSQL`
- running the requested background process
- updating `PostgreSQL`
- acking or rejecting messages based on processing outcome

## RabbitMQ settings

Default local settings:
- URL: `amqp://guest:guest@localhost:5672/`
- meal-analysis queue: `foodsnap.meal_analysis`
- food-reference import queue: `foodsnap.food_reference_import`
- queue durability: durable
- message delivery mode: persistent
- content type: `application/json`

Environment variables:
- `QUEUE_BACKEND=rabbitmq`
- `RABBITMQ_URL=amqp://guest:guest@localhost:5672/`
- `RABBITMQ_MEAL_ANALYSIS_QUEUE=foodsnap.meal_analysis`
- `RABBITMQ_FOOD_REFERENCE_IMPORT_QUEUE=foodsnap.food_reference_import`

## Meal-analysis message

The API publishes this JSON message after a meal entry is created:

```json
{
  "event_type": "meal.analysis.requested",
  "version": 1,
  "meal_id": "meal_abc123",
  "occurred_at": "2026-04-15T12:00:00+00:00"
}
```

Fields:
- `event_type`: stable event name for routing and debugging
- `version`: message contract version
- `meal_id`: canonical meal id in `PostgreSQL`
- `occurred_at`: UTC timestamp generated by the API

## Food-reference import message

The API publishes this JSON message after `POST /api/v1/food-reference/imports`:

```json
{
  "event_type": "food_reference.import.requested",
  "version": 1,
  "import_request_id": "food_import_abc123",
  "source": "usda_fdc",
  "labels": ["pizza", "salad", "burger"],
  "limit_per_label": 3,
  "mode": "upsert",
  "requested_by_user_id": "user_abc123",
  "occurred_at": "2026-04-15T12:00:00+00:00"
}
```

Fields:
- `event_type`: stable event name for routing and debugging
- `version`: message contract version
- `import_request_id`: generated id for tracing this import request
- `source`: external provider, currently `usda_fdc` or `open_food_facts`
- `labels`: normalized dish labels to search and import
- `limit_per_label`: max external records to inspect per label
- `mode`: import behavior, currently `upsert` or `insert_missing`
- `requested_by_user_id`: user who requested the import
- `occurred_at`: UTC timestamp generated by the API

## Expected worker flow

The separate meal-analysis worker microservice should roughly do this:

1. connect to RabbitMQ
2. declare the same durable queue
3. consume messages with manual acknowledgement
4. parse JSON and validate `event_type`, `version`, and `meal_id`
5. mark the meal as `processing`
6. run recognition and calorie estimation
7. mark the meal as `done`
8. ack the RabbitMQ message

On processing failure:
- mark the meal as `failed` when the failure is business-level or non-retriable
- reject/nack the message based on retry strategy
- avoid acking a failed transient task before it is safely handled

The separate food-reference import worker microservice should roughly do this:

1. connect to RabbitMQ
2. declare `foodsnap.food_reference_import`
3. consume messages with manual acknowledgement
4. validate `event_type`, `version`, `source`, and `labels`
5. call the configured external food data provider
6. normalize external records into internal `food_reference` rows
7. upsert or insert missing records depending on `mode`
8. ack the RabbitMQ message

## Why queue is needed

Photo analysis is not a good fit for synchronous HTTP request handling.

The queue lets the API:
- return quickly after upload
- avoid HTTP timeouts during image analysis
- isolate API scaling from worker scaling
- retry failed processing independently
- keep meal status visible through `pending`, `processing`, `done`, and `failed`

## Local startup

Start RabbitMQ locally:

```bash
docker compose up -d rabbitmq
```

RabbitMQ management UI:

```text
http://localhost:15672
```

Default credentials:
- username: `guest`
- password: `guest`
